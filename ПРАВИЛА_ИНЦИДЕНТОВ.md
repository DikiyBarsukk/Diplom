# Правила выявления инцидентов BARSUKSIEM

## Обзор

В проекте реализован rule-based анализатор инцидентов в `server/incidents.py`.

Текущие правила:

- `R001` - brute force attack detection
- `R002` - unauthorized access / night admin login
- `R003` - suspicious activity / PowerShell + download
- `R004` - log tampering detection
- `R005` - privilege escalation detection

## R001 - Brute Force Attack Detection

- тип инцидента: `brute_force`
- severity: `high`
- идея: 5 и более неудачных попыток входа за 10 минут
- ожидаемый эффект: создается инцидент по хосту с указанием временного окна и связанных событий

## R002 - Unauthorized Access - Night Admin Login

- тип инцидента: `unauthorized_access`
- severity: `medium`
- идея: вход администратора в интервале `00:00-06:00`
- ожидаемый эффект: инцидент с признаком аномального времени входа

## R003 - Suspicious Activity - PowerShell Download

- тип инцидента: `suspicious_activity`
- severity: `high`
- идея: запуск PowerShell вместе с сетевой загрузкой
- ожидаемый эффект: инцидент, указывающий на возможную вредоносную активность

## R004 - Log Tampering Detection

- тип инцидента: `log_tampering`
- severity зависит от правила, обычно повышенная
- идея: признаки удаления журналов, очистки следов или вмешательства в логирование
- ожидаемый эффект: инцидент по попытке сокрытия следов

## R005 - Privilege Escalation Detection

- тип инцидента: `privilege_escalation`
- severity: `critical` или `high` в зависимости от контекста правила
- идея: корреляция цепочки failed login -> success -> privileged login
- ожидаемый эффект: инцидент с корреляционным паттерном повышения привилегий

## Данные на вход анализатора

Анализатор работает по нормализованным событиям из `server/parser.py` и использует поля:

- `ts`
- `host`
- `message`
- `process`
- `unit`
- `severity`
- `id` при наличии

## Что проверено тестами

Unit tests покрывают:

- `R001` - достижение порога brute force
- `R002` - обнаружение ночного логина администратора
- `R003` - корреляцию подозрительной загрузки через PowerShell

## Практическая демонстрация на защите

Для защиты удобно показать как минимум 2-3 сценария:

1. серия неудачных логинов
2. ночной вход администратора
3. suspicious PowerShell activity

После ingest этих demo-событий инциденты становятся видимыми в `/api/incidents` и `/incidents`.
