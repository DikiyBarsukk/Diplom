# Размещение файлов и эксплуатация BARSUKSIEM

## 1. Назначение документа

Этот документ описывает:

- какие каталоги и файлы нужны на серверном узле;
- какие каталоги и файлы нужны на агенте;
- какие каталоги и файлы нужны на desktop-клиенте;
- какие файлы можно не переносить на production-узлы;
- как запускать, обновлять, проверять и обслуживать систему;
- какие журналы, настройки и переменные окружения контролировать в эксплуатации.

Документ рассчитан на развертывание проекта как на одной машине для демонстрации, так и на нескольких отдельных узлах.

## 2. Варианты размещения

### Вариант A. Один узел для демонстрации

На одной машине находятся:

- сервер FastAPI;
- web UI;
- SQLite база;
- агент;
- desktop-клиент.

Используется для дипломной демонстрации, локальной отладки и smoke-проверок.

### Вариант B. Разделенное размещение

Используются отдельные роли:

- серверный узел: API, web UI, база данных, правила инцидентов;
- один или несколько агентных узлов: сбор логов и отправка на сервер;
- рабочее место аудитора: desktop-клиент и web UI через браузер.

Это основной рекомендуемый вариант для эксплуатации.

## 3. Структура проекта в репозитории

Корень проекта содержит:

```text
ПО/
├─ main.py
├─ requirements-server.txt
├─ requirements-client.txt
├─ server/
├─ client/
├─ common/
├─ web/
├─ tests/
├─ scripts/
└─ *.md
```

Назначение верхнего уровня:

- `main.py` - единая CLI-точка входа для режимов `server`, `agent`, `client`.
- `requirements-server.txt` - зависимости серверного узла.
- `requirements-client.txt` - зависимости агента и desktop-клиента.
- `server/` - серверная логика, API, auth, storage, incident detection.
- `client/` - агент и desktop-клиент.
- `common/` - общие утилиты для чтения логов и системных источников.
- `web/` - HTML-страницы и статические JS/CSS-ресурсы web UI.
- `tests/` - автоматические тесты и demo-данные.
- `scripts/` - вспомогательные скрипты, если используются в проекте.

## 4. Какие файлы должны быть на сервере

### 4.1 Рекомендуемый состав серверного узла

На серверный узел рекомендуется переносить:

```text
server-node/
├─ main.py
├─ requirements-server.txt
├─ server/
│  ├─ app_factory.py
│  ├─ auth.py
│  ├─ config.py
│  ├─ storage.py
│  ├─ parser.py
│  ├─ incidents.py
│  ├─ analyzer.py
│  ├─ security.py
│  ├─ cache.py
│  ├─ time_utils.py
│  ├─ fetcher.py
│  └─ routes/
├─ common/
│  ├─ __init__.py
│  └─ log_utils.py
├─ web/
│  ├─ login.html
│  ├─ index.html
│  ├─ logs.html
│  ├─ incidents.html
│  ├─ incident_details.html
│  ├─ analytics.html
│  ├─ compliance.html
│  ├─ remote_pcs.html
│  └─ static/
└─ barsuksiem.db
```

### 4.2 Что делает каждый серверный каталог

- `server/app_factory.py` - собирает FastAPI-приложение через `create_app()`.
- `server/routes/` - регистрирует page routes и `/api/...` маршруты.
- `server/auth.py` - login/logout/session/csrf/bootstrap admin/audit log.
- `server/storage.py` - операции с SQLite.
- `server/parser.py` - нормализация событий перед записью.
- `server/incidents.py` - выявление инцидентов на основе правил.
- `server/security.py` - rate limiting, security headers, CSRF helpers.
- `server/config.py` - читает env-конфигурацию.
- `web/` - клиентская часть браузерного интерфейса.
- `common/log_utils.py` - общие функции, которые также нужны части серверных сценариев.

### 4.3 Что не обязательно держать на сервере

Необязательно переносить на production-сервер:

- `tests/`
- `.idea/`
- `.venv/`
- markdown-документы, если они не нужны на узле
- desktop GUI-файлы, если на сервере не запускается `python main.py client`

### 4.4 Минимально необходимый состав для сервера

Если нужен именно минимальный пакет, а не полная копия репозитория, достаточно:

- `main.py`
- `requirements-server.txt`
- весь каталог `server/`
- каталог `web/`
- каталог `common/`

## 5. Какие файлы должны быть на агенте

### 5.1 Рекомендуемый состав агентного узла

```text
agent-node/
├─ main.py
├─ requirements-client.txt
├─ client/
│  ├─ agent.py
│  ├─ encryption.py
│  ├─ udp_client.py
│  ├─ connection.py
│  └─ gui.py
└─ common/
   ├─ __init__.py
   └─ log_utils.py
```

### 5.2 Что реально нужно агенту

Для запуска `python main.py agent ...` используются:

- `main.py`
- `client/agent.py`
- `client/encryption.py`
- `client/udp_client.py`
- `common/log_utils.py`
- `requirements-client.txt`

### 5.3 Что агенту не нужно

Агентному узлу не требуются:

- `server/`
- `web/`
- `tests/`
- `client/gui.py`, если desktop GUI на этом узле не используется
- `client/connection.py`, если агент не совмещен с desktop-клиентом

### 5.4 Источники данных агента

Агент может работать с:

- `--source auto`
- `--source journal`
- `--source file`

Вариант выбирается по среде и доступным источникам.

## 8. Подготовка серверного узла

### 8.1 Требования

- Python 3.11+
- доступный TCP-порт для API и web UI
- файловая система с правом записи для SQLite базы
- сетевой доступ от агентов к серверу

### 8.2 Установка зависимостей

```bash
python -m pip install -U pip
python -m pip install -U -r requirements-server.txt
```

### 8.3 Рекомендуемые переменные окружения сервера

Windows:

```powershell
set BARSUKSIEM_BOOTSTRAP_ADMIN_USERNAME=admin
set BARSUKSIEM_BOOTSTRAP_ADMIN_PASSWORD=StrongPassword123!
set BARSUKSIEM_COOKIE_SECURE=false
set BARSUKSIEM_COOKIE_SAMESITE=lax
set BARSUKSIEM_SESSION_MAX_AGE=86400
set CORS_ALLOWED_ORIGINS=http://127.0.0.1:8080,http://localhost:8080
```

Linux/macOS:

```bash
export BARSUKSIEM_BOOTSTRAP_ADMIN_USERNAME=admin
export BARSUKSIEM_BOOTSTRAP_ADMIN_PASSWORD=StrongPassword123!
export BARSUKSIEM_COOKIE_SECURE=false
export BARSUKSIEM_COOKIE_SAMESITE=lax
export BARSUKSIEM_SESSION_MAX_AGE=86400
export CORS_ALLOWED_ORIGINS=http://127.0.0.1:8080,http://localhost:8080
```

### 8.4 Запуск сервера

```bash
python main.py server --host 0.0.0.0 --port 8080
```

Для HTTPS:

```bash
python main.py server --host 0.0.0.0 --port 8443 --ssl-cert cert.pem --ssl-key key.pem
```

### 8.5 Что проверить после старта

- сервер отвечает на `/health`;
- web UI открывается по `/login`;
- в логе запуска нет ошибок инициализации;
- при пустой БД создан bootstrap admin;
- база SQLite создана и доступна для записи.

## 9. Подготовка агентного узла

### 9.1 Установка зависимостей

```bash
python -m pip install -U pip
python -m pip install -U -r requirements-client.txt
```

### 9.2 Что проверить перед запуском агента

- сервер доступен по сети;
- URL сервера указан как `http://host:port`, а не HTML-страница;
- локальный источник логов доступен текущему пользователю;
- время на агенте синхронизировано;
- firewall не блокирует исходящее соединение.

### 9.3 Запуск агента

Автовыбор источника:

```bash
python main.py agent --server http://127.0.0.1:8080 --source auto --interval 60
```

Journal:

```bash
python main.py agent --server http://127.0.0.1:8080 --source journal --interval 60
```

Файловый источник:

```bash
python main.py agent --server http://127.0.0.1:8080 --source file --interval 60
```

UDP/TCP:

```bash
python main.py agent --server udp://127.0.0.1:9000 --source auto
python main.py agent --server tcp://127.0.0.1:9001 --source auto
```

### 9.4 Что проверить после старта агента

- на сервере появились новые события в `/api/logs`;
- в web UI виден ingest;
- статистика и инциденты обновляются;
- у агента нет циклических ошибок отправки.

## 11. Порядок запуска системы в эксплуатации

Рекомендуемый порядок:

1. Поднять сервер.
2. Убедиться, что `/health` отвечает.
3. Выполнить вход в `/login` под bootstrap admin или штатной учетной записью.
4. Поднять один или несколько агентов.
5. Убедиться, что логи поступают в `/logs`.
6. Проверить, что инциденты видны в `/incidents`.
7. При необходимости запустить desktop-клиент аудитора.

## 13. Аутентификация и сессии в эксплуатации

### 13.1 Как работает вход

- пользователь входит через `/api/auth/login`;
- сервер создает session cookie;
- web UI и desktop-клиент работают в рамках этой сессии;
- для изменяющих операций используется `X-CSRF-Token`.

### 13.2 Как получать CSRF token

- использовать `GET /api/auth/me`;
- сервер возвращает текущего пользователя;
- актуальный токен считывается из response header.

### 13.3 Что важно для эксплуатации

- не отключать cookie в браузере;
- не использовать hardcoded admin/password;
- хранить bootstrap-пароль отдельно от публичной документации;
- после первичной настройки создать штатную учетную запись администратора.

## 14. Повседневная эксплуатация

### 14.1 Ежедневный чек-лист оператора

- убедиться, что сервер отвечает на `/health`;
- проверить наличие новых логов;
- проверить открытые инциденты;
- убедиться, что последний ingest свежий;
- проверить, что критические хосты видны в dashboard.

### 14.2 Еженедельный чек-лист администратора

- сохранить резервную копию SQLite базы;
- проверить срок жизни bootstrap-учетки и наличие штатных учетных записей;
- проверить настройки `CORS_ALLOWED_ORIGINS`, cookie и session age;
- проверить объем базы и рост журналов;
- обновить зависимости в тестовом контуре при необходимости.

### 14.3 Что делать перед демонстрацией комиссии

- запустить сервер заранее;
- проверить bootstrap/admin учетную запись;
- подготовить тестовые события или demo ingest;
- убедиться, что `/logs`, `/incidents`, `/analytics` открываются без ошибок;
- подготовить короткий smoke-сценарий;
- прогнать `python -m unittest discover -s tests -v`.

## 15. Резервное копирование

### 15.1 Что обязательно резервировать

- SQLite базу (`barsuksiem.db` или фактический путь к БД);
- env-файл или набор переменных окружения;
- TLS-сертификаты, если включен HTTPS;
- локальные эксплуатационные инструкции и учетные данные доступа.

### 15.2 Простой подход

1. Остановить серверный процесс.
2. Скопировать файл базы в каталог резервных копий.
3. Скопировать конфигурационные env-данные.
4. Зафиксировать дату, версию кода и ответственного.

## 16. Обновление системы

### 16.1 Безопасный порядок обновления

1. Сделать резервную копию БД.
2. Сохранить текущие env-настройки.
3. Обновить код проекта.
4. Переустановить зависимости из соответствующего requirements-файла.
5. Запустить автоматические тесты.
6. Поднять сервер и проверить `/health`.
7. Проверить login, ingest, `/logs`, `/incidents`, `/analytics`.

### 16.2 Что важно не потерять

- файл БД;
- env-переменные;
- bootstrap/admin доступ;
- совместимость agent URL и серверного API.

## 17. Чего не должно быть на production-узле

Не рекомендуется:

- держать пароль администратора в README или открытых bat/sh-файлах;
- публиковать `.venv/` в составе артефакта;
- использовать старые маршруты вместо `/api/...` в интеграциях;
- смешивать HTML-роуты и программные API в клиентском коде;
- оставлять демонстрационные учетные данные в постоянной эксплуатации.

## 18. Быстрый сценарий эксплуатации после установки

1. Установить зависимости на сервере.
2. Задать env-переменные сервера.
3. Запустить `python main.py server --host 0.0.0.0 --port 8080`.
4. Открыть `/login` и войти под bootstrap admin.
5. На агенте установить client-зависимости.
6. Запустить `python main.py agent --server http://SERVER:8080 --source auto --interval 60`.
7. Проверить появление логов в `/logs`.
8. Проверить обнаружение инцидентов в `/incidents`.
9. При необходимости запустить GUI: `python main.py client --server http://SERVER:8080`.
10. Зафиксировать рабочую конфигурацию.

## 19. Типовые проблемы и действия

### 19.1 Сервер запустился, но login не работает

Проверьте:

- что БД доступна для записи;
- что bootstrap admin создан;
- что пароль берется из актуального env или startup log;
- что браузер не блокирует cookie.

### 19.2 Агент не передает данные

Проверьте:

- URL сервера;
- доступность `/health`;
- исходящий firewall;
- права на чтение journal или файла логов.

### 19.3 Web UI открывается, но POST-запросы падают

Проверьте:

- есть ли session cookie;
- отправляется ли `X-CSRF-Token`;
- не истекла ли сессия;
- нет ли конфликта между HTTP/HTTPS и `BARSUKSIEM_COOKIE_SECURE`.

### 19.4 Desktop-клиент не подключается

Проверьте:

- `BARSUKSIEM_CLIENT_USERNAME`;
- `BARSUKSIEM_CLIENT_PASSWORD`;
- правильность `--server`;
- доступность `/api/auth/login` и `/api/stats`.
