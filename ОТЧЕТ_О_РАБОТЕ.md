# 📋 Отчет о проделанной работе

## Система централизованного аудита логов

---

## 1. Описание системы

### 1.1. Назначение

Разработана система централизованного сбора, хранения и анализа логов с различных хостов в едином хранилище. Система обеспечивает:

- Централизованный сбор логов с множества машин
- Нормализацию событий из разных источников (Linux journalctl, Windows Event Log, файлы)
- Хранение логов в базе данных SQLite
- Визуализацию и анализ логов через графический интерфейс
- Фильтрацию и поиск по различным критериям
- Статистический анализ и построение графиков

### 1.2. Архитектура системы

Система построена по клиент-серверной архитектуре и состоит из трех компонентов:

```
┌─────────────────────────────────────────────────────────┐
│                    АГЕНТЫ (N машин)                     │
│  - Сбор логов локально                                  │
│  - Отправка на сервер                                   │
│  - Поддержка: Linux, Windows                            │
└───────────────────┬─────────────────────────────────────┘
                    │ HTTP POST /logs
                    ▼
┌─────────────────────────────────────────────────────────┐
│                    СЕРВЕР (1 машина)                    │
│  - Прием логов от агентов                               │
│  - Нормализация событий                                 │
│  - Хранение в SQLite                                    │
│  - REST API (FastAPI)                                   │
└───────────────────┬─────────────────────────────────────┘
                    │ HTTP GET /logs, /stats
                    ▼
┌─────────────────────────────────────────────────────────┐
│              КЛИЕНТ-АУДИТОР (1 машина)                  │
│  - GUI на PySide6                                       │
│  - Просмотр и фильтрация логов                          │
│  - Статистика и графики                                 │
└─────────────────────────────────────────────────────────┘
```

### 1.3. Технологический стек

**Backend:**
- Python 3.10+
- FastAPI — веб-фреймворк для REST API
- Uvicorn — ASGI сервер
- SQLite — база данных для хранения логов
- Requests — HTTP клиент

**Frontend:**
- PySide6 (Qt6) — графический интерфейс
- QtCharts — построение графиков и диаграмм

**Системные зависимости:**
- journalctl (Linux) — сбор системных логов
- Windows Event Log API — сбор логов Windows
- PowerShell (Windows) — альтернативный метод сбора

## 2. Реализованные функции

### 2.1. Сервер (server/)

#### 2.1.1. REST API (main.py, server/)

**Реализованные эндпоинты:**

1. **GET /health** — проверка доступности сервера
   - Возвращает статус, hostname и версию

2. **POST /logs** — прием логов от агентов
   - Принимает массив сырых событий
   - Нормализует события
   - Сохраняет в базу данных
   - Возвращает количество сохраненных и пропущенных событий

3. **GET /logs** — получение логов с фильтрацией
   - Параметры фильтрации:
     - `host` — фильтр по hostname
     - `severity` — фильтр по уровню важности
     - `since` — фильтр по времени (ISO формат)
     - `search` — поиск по содержимому сообщения (LIKE)
     - `limit` — максимальное количество событий (1-1000)
     - `offset` — смещение для пагинации
   - Возвращает отсортированный по времени список событий

4. **GET /stats** — статистика по логам
   - Общее количество событий
   - Распределение по хостам
   - Распределение по уровням важности
   - Время последнего события

#### 2.1.2. Нормализация событий (server/parser.py)

**Функция `normalize_event()`:**
- Унифицирует события из разных источников в единый формат
- Извлекает и нормализует:
  - Timestamp (поддержка различных форматов)
  - Hostname
  - Source (journal, eventlog, file)
  - Unit/Service (systemd unit или Windows service)
  - Process name
  - PID (Process ID)
  - UID (User ID)
  - Severity (уровень важности)
  - Message (текст сообщения)
  - Raw data (оригинальное событие)
- Генерирует уникальный hash для предотвращения дубликатов

**Поддерживаемые источники:**
- Linux journalctl (__REALTIME_TIMESTAMP, PRIORITY, _SYSTEMD_UNIT, и т.д.)
- Windows Event Log (TimeGenerated, EventType, SourceName, и т.д.)
- Файлы логов (системные файлы /var/log/*)

**Маппинг приоритетов:**
- Linux syslog priority (0-7) → стандартные уровни (emerg, alert, crit, err, warn, notice, info, debug)
- Windows Event Log levels (Critical, Error, Warning, Information) → стандартные уровни

#### 2.1.3. Хранение данных (server/storage.py)

**Класс `LogStorage`:**
- Управление базой данных SQLite
- Автоматическое создание таблиц и индексов
- Дедупликация по hash

**Структура таблицы `events`:**
```sql
CREATE TABLE events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    hash TEXT UNIQUE,           -- Уникальный hash для дедупликации
    ts TEXT NOT NULL,           -- Timestamp события
    host TEXT NOT NULL,         -- Hostname источника
    source TEXT NOT NULL,       -- Источник (journal, eventlog, file)
    unit TEXT,                  -- Systemd unit / Service
    process TEXT,               -- Имя процесса
    pid INTEGER,                -- Process ID
    uid INTEGER,                -- User ID
    severity TEXT NOT NULL,     -- Уровень важности
    message TEXT NOT NULL,      -- Текст сообщения
    raw_data TEXT,              -- Оригинальное событие (JSON)
    ingest_ts TEXT NOT NULL,    -- Время получения сервером
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
)
```

**Индексы для оптимизации:**
- `idx_host` — быстрый поиск по хосту
- `idx_ts` — быстрая сортировка по времени
- `idx_severity` — быстрая фильтрация по уровню важности
- `idx_hash` — быстрая проверка дубликатов

**Методы:**
- `store_event()` — сохранение одного события (с обработкой дубликатов)
- `store_events()` — массовое сохранение событий
- `get_events()` — получение событий с фильтрацией
- `get_stats()` — получение статистики

#### 2.1.4. Анализ данных (server/analyzer.py)

- Функция `severity_counts()` — подсчет событий по уровням важности
- Используется для статистики и графиков

### 2.2. Агент (client/agent.py)

#### 2.2.1. Сбор логов

**Функция `collect_logs()`:**
- Автоматическое определение платформы и источника логов
- Поддержка источников:
  - `auto` — автоматическое определение
  - `journal` — Linux journalctl
  - `eventlog` — Windows Event Log
  - `file` — файлы логов (Linux)

**Методы сбора:**

1. **Linux journalctl** (`iter_journal_json()`):
   - Использует `journalctl -o json`
   - Поддержка фильтрации по времени (`--since`)
   - Ограничение количества (`-n`)

2. **Windows Event Log** (`get_windows_event_logs()`):
   - Основной метод: `win32evtlog` через pywin32
   - Альтернативный метод: PowerShell `Get-WinEvent`
   - Чтение журналов: System, Application, Security
   - Поддержка фильтрации по времени

3. **Файлы логов** (`read_file_tail()`):
   - Чтение последних строк из:
     - `/var/log/syslog`
     - `/var/log/messages`
     - `/var/log/auth.log`
   - Обработка ошибок кодировки

#### 2.2.2. Отправка на сервер

**Функция `run_agent()` (main.py):**
- Периодический сбор логов (интервал настраивается)
- Отправка на сервер через HTTP POST
- Обработка ошибок подключения
- Логирование операций
- Обработка прерывания (Ctrl+C)

**Параметры агента:**
- `--server` — URL сервера (обязательный)
- `--source` — источник логов (auto, journal, eventlog, file)
- `--limit` — количество событий за цикл (по умолчанию 200)
- `--interval` — интервал отправки в секундах (по умолчанию 60)

### 2.3. Клиент-аудитор (client/gui.py)

#### 2.3.1. Графический интерфейс

**Основное окно:**
- Двухвкладочный интерфейс (Logs, Dashboard)
- Адаптивный размер окна (1400x800)

#### 2.3.2. Вкладка "Logs" — Просмотр логов

**Фильтры:**
1. **Host** — фильтр по hostname (текстовое поле)
2. **Severity** — фильтр по уровню важности (выпадающий список: emerg, alert, crit, err, warn, notice, info, debug)
3. **Search** — поиск по содержимому сообщения (текстовое поле, поиск по Enter)
4. **Time Filter** — фильтр по времени:
   - All time — все время
   - Last hour — последний час
   - Last 6 hours — последние 6 часов
   - Last 24 hours — последние 24 часа
   - Last 7 days — последние 7 дней
   - Custom... — произвольная дата и время
5. **Limit** — количество отображаемых событий (1-1000, по умолчанию 200)

**Таблица логов:**
- Отображение колонок: ts, severity, host, unit, pid, message
- Сортировка по времени (по умолчанию — новые сверху)
- Растягивание последней колонки (message)
- Выбор строк
- **Двойной клик** — открытие окна с детальной информацией о событии

**Детальное окно события:**
- Отображение всех полей:
  - Timestamp
  - Host
  - Source
  - Severity
  - Unit
  - Process
  - PID
  - UID
  - Message
  - Raw Data (оригинальное событие)

**Кнопки:**
- **Fetch** — загрузка логов с применением фильтров
- **Dashboard** — переключение на вкладку Dashboard
- **Stats** — отображение статистики (в статусной строке)

#### 2.3.3. Вкладка "Dashboard" — Аналитика

**Графики:**

1. **Events by Severity** (круговая диаграмма):
   - Распределение событий по уровням важности
   - Цветовая индикация:
     - emerg, alert, crit, err — оттенки красного/оранжевого
     - warn — оранжевый
     - notice — золотой
     - info — синий
     - debug — серый
   - Легенда справа

2. **Events by Host (Top 10)** (столбчатая диаграмма):
   - Топ-10 хостов по количеству событий
   - Горизонтальная гистограмма
   - Подписи хостов (ограничение длины 20 символов)

3. **Events Timeline (Last 24 hours)** (линейный график):
   - Временная линия событий за последние 24 часа
   - Группировка по часам
   - Ось X — время (формат HH:mm)
   - Ось Y — количество событий

**Кнопки:**
- **Refresh Dashboard** — обновление всех графиков

#### 2.3.4. Модель данных (LogsTableModel)

- Наследование от `QAbstractTableModel`
- Динамическое обновление данных
- Поддержка стандартных методов Qt для работы с таблицами

#### 2.3.5. Клиент для связи с сервером (client/connection.py)

**Класс `ServerClient`:**
- Инкапсуляция HTTP запросов
- Методы:
  - `health()` — проверка доступности сервера
  - `fetch_logs()` — получение логов с фильтрацией
  - `get_stats()` — получение статистики
  - `send_logs()` — отправка логов (используется агентом)
- Обработка таймаутов
- Обработка ошибок HTTP
