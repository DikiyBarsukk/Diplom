# 📋 Отчет о проделанной работе

## Система централизованного аудита логов

---

## 1. Описание системы

### 1.1. Назначение

Разработана система централизованного сбора, хранения и анализа логов с различных хостов в едином хранилище. Система обеспечивает:

- Централизованный сбор логов с множества машин
- Нормализацию событий из разных источников (Linux journalctl, Windows Event Log, файлы)
- Хранение логов в базе данных SQLite
- Визуализацию и анализ логов через графический интерфейс
- Фильтрацию и поиск по различным критериям
- Статистический анализ и построение графиков

### 1.2. Архитектура системы

Система построена по клиент-серверной архитектуре и состоит из трех компонентов:

```
┌─────────────────────────────────────────────────────────┐
│                    АГЕНТЫ (N машин)                     │
│  - Сбор логов локально                                  │
│  - Отправка на сервер                                   │
│  - Поддержка: Linux, Windows                            │
└───────────────────┬─────────────────────────────────────┘
                    │ HTTP POST /logs
                    ▼
┌─────────────────────────────────────────────────────────┐
│                    СЕРВЕР (1 машина)                    │
│  - Прием логов от агентов                               │
│  - Нормализация событий                                 │
│  - Хранение в SQLite                                    │
│  - REST API (FastAPI)                                   │
└───────────────────┬─────────────────────────────────────┘
                   │ HTTP GET /api/logs, /stats
                    ▼
┌─────────────────────────────────────────────────────────┐
│              КЛИЕНТ-АУДИТОР (1 машина)                  │
│  - GUI на PySide6                                       │
│  - Просмотр и фильтрация логов                          │
│  - Статистика и графики                                 │
└─────────────────────────────────────────────────────────┘
```

### 1.3. Технологический стек

**Backend:**
- Python 3.10+
- FastAPI — веб-фреймворк для REST API
- Uvicorn — ASGI сервер
- SQLite — база данных для хранения логов
- Requests — HTTP клиент

**Frontend:**
- PySide6 (Qt6) — графический интерфейс
- QtCharts — построение графиков и диаграмм

**Системные зависимости:**
- journalctl (Linux) — сбор системных логов
- Windows Event Log API — сбор логов Windows
- PowerShell (Windows) — альтернативный метод сбора

## 2. Реализованные функции

### 2.1. Сервер (server/)

#### 2.1.1. REST API (main.py, server/)

**Реализованные эндпоинты:**

1. **GET /health** — проверка доступности сервера
   - Возвращает статус, hostname и версию

2. **POST /logs** — прием логов от агентов
   - Принимает массив сырых событий
   - Нормализует события
   - Сохраняет в базу данных
   - Возвращает количество сохраненных и пропущенных событий

3. **GET /api/logs** — получение логов с фильтрацией
   - Параметры фильтрации:
     - `host` — фильтр по hostname
     - `severity` — фильтр по уровню важности
     - `since` — фильтр по времени (ISO формат)
     - `search` — поиск по содержимому сообщения (LIKE)
    - `limit` — максимальное количество событий (1-10000)
     - `offset` — смещение для пагинации
   - Возвращает отсортированный по времени список событий

4. **GET /stats** — статистика по логам
   - Общее количество событий
   - Распределение по хостам
   - Распределение по уровням важности
   - Время последнего события
   - **Кэширование:** результаты кэшируются на 5 минут для повышения производительности

5. **GET /api/agents/stats** — статистика по агентам (онлайн/оффлайн)
   - Возвращает количество онлайн/оффлайн за заданное окно

6. **POST /api/auth/login** — аутентификация пользователя
   - Принимает username и password
   - Возвращает токен сессии
   - Устанавливает cookie `session_token`
   - Логирует попытки входа (успешные и неудачные)

7. **POST /api/auth/logout** — выход из системы
   - Удаляет сессию пользователя
   - Требует аутентификации

8. **GET /api/auth/me** — информация о текущем пользователе
   - Возвращает username и role
   - Требует аутентификации

9. **GET /api/audit** — журнал аудита действий пользователей
   - Показывает все действия: входы, выходы, просмотры логов
   - Только для пользователей с ролью `admin`
   - Поддерживает пагинацию (limit, offset)

#### 2.1.2. Нормализация событий (server/parser.py)

**Функция `normalize_event()`:**
- Унифицирует события из разных источников в единый формат
- Извлекает и нормализует:
  - Timestamp (поддержка различных форматов)
  - Hostname
  - Source (journal, eventlog, file)
  - Unit/Service (systemd unit или Windows service)
  - Process name
  - PID (Process ID)
  - UID (User ID)
  - Severity (уровень важности)
  - Message (текст сообщения)
  - Raw data (оригинальное событие)
- Генерирует уникальный hash для предотвращения дубликатов

**Поддерживаемые источники:**
- Linux journalctl (__REALTIME_TIMESTAMP, PRIORITY, _SYSTEMD_UNIT, и т.д.)
- Windows Event Log (TimeGenerated, EventType, SourceName, и т.д.)
- Файлы логов (системные файлы /var/log/*)

**Маппинг приоритетов:**
- Linux syslog priority (0-7) → стандартные уровни (emerg, alert, crit, err, warn, notice, info, debug)
- Windows Event Log levels (Critical, Error, Warning, Information) → стандартные уровни

#### 2.1.3. Хранение данных (server/storage.py)

**Класс `LogStorage`:**
- Управление базой данных SQLite
- Автоматическое создание таблиц и индексов
- Дедупликация по hash

**Структура таблицы `events`:**
```sql
CREATE TABLE events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    hash TEXT UNIQUE,           -- Уникальный hash для дедупликации
    ts TEXT NOT NULL,           -- Timestamp события
    host TEXT NOT NULL,         -- Hostname источника
    source TEXT NOT NULL,       -- Источник (journal, eventlog, file)
    unit TEXT,                  -- Systemd unit / Service
    process TEXT,               -- Имя процесса
    pid INTEGER,                -- Process ID
    uid INTEGER,                -- User ID
    severity TEXT NOT NULL,     -- Уровень важности
    message TEXT NOT NULL,      -- Текст сообщения
    raw_data TEXT,              -- Оригинальное событие (JSON)
    ingest_ts TEXT NOT NULL,    -- Время получения сервером
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
)
```

**Индексы для оптимизации:**
- `idx_host` — быстрый поиск по хосту
- `idx_ts` — быстрая сортировка по времени
- `idx_severity` — быстрая фильтрация по уровню важности
- `idx_hash` — быстрая проверка дубликатов

**Дополнительные таблицы (версия 0.3):**
- `users` — пользователи системы (username, password_hash, role)
- `sessions` — активные сессии пользователей (token, expires_at)
- `audit_log` — журнал действий пользователей (action, resource, details, ip_address)

**Методы:**
- `store_event()` — сохранение одного события (с обработкой дубликатов)
- `store_events()` — массовое сохранение событий
- `get_events()` — получение событий с фильтрацией
- `get_stats()` — получение статистики

#### 2.1.4. Аутентификация и авторизация (server/auth.py)

**Класс `AuthManager`:**
- Управление пользователями и сессиями
- Хеширование паролей (SHA-256)
- Генерация токенов сессий
- Проверка разрешений по ролям

**Роли пользователей:**
- `admin` — полный доступ (read, write, delete, manage_users)
- `auditor` — только чтение (read)
- `guest` — только чтение (read)

**Функции:**
- `authenticate()` — проверка учетных данных
- `create_session()` — создание сессии (TTL 24 часа)
- `validate_session()` — проверка валидности сессии
- `logout()` — удаление сессии
- `has_permission()` — проверка разрешений
- `log_action()` — логирование действий пользователей
- `get_audit_log()` — получение журнала аудита
- `create_user()` — создание нового пользователя
- `list_users()` — список всех пользователей

**Пользователь по умолчанию:**
- Username: `admin`
- Password: `Admin123!@#` (соответствует политике безопасности)
- Role: `admin`

#### 2.1.5. Кэширование (server/cache.py)

**Класс `Cache`:**
- In-memory кэш с TTL (Time To Live)
- Автоматическая очистка истекших записей
- Используется для кэширования результатов запросов

**Применение:**
- Кэширование статистики (`/stats`) — TTL 5 минут
- Кэширование результатов поиска (`/api/logs`) — TTL 60 секунд
- Автоматическая очистка кэша при добавлении новых логов

#### 2.1.6. Анализ данных (server/analyzer.py)

- Функция `severity_counts()` — подсчет событий по уровням важности
- Используется для статистики и графиков

### 2.2. Агент (client/agent.py)

#### 2.2.1. Сбор логов

**Функция `collect_logs()`:**
- Автоматическое определение платформы и источника логов
- Поддержка источников:
  - `auto` — автоматическое определение
  - `journal` — Linux journalctl
  - `eventlog` — Windows Event Log
  - `file` — файлы логов (Linux)

**Методы сбора:**

1. **Linux journalctl** (`iter_journal_json()`):
   - Использует `journalctl -o json`
   - Поддержка фильтрации по времени (`--since`)
   - Ограничение количества (`-n`)

2. **Windows Event Log** (`get_windows_event_logs()`):
   - Основной метод: `win32evtlog` через pywin32
   - Альтернативный метод: PowerShell `Get-WinEvent`
   - Чтение журналов: System, Application, Security
   - Поддержка фильтрации по времени

3. **Файлы логов** (`read_file_tail()`):
   - Чтение последних строк из:
     - `/var/log/syslog`
     - `/var/log/messages`
     - `/var/log/auth.log`
   - Обработка ошибок кодировки

#### 2.2.2. Отправка на сервер

**Функция `run_agent()` (main.py):**
- Периодический сбор логов (интервал настраивается)
- Отправка на сервер через HTTP POST
- Обработка ошибок подключения
- Логирование операций
- Обработка прерывания (Ctrl+C)

**Параметры агента:**
- `--server` — URL сервера (обязательный)
  - Поддерживает протоколы: `http://`, `https://`, `udp://`, `tcp://`
  - Примеры:
    - `http://192.168.1.100:8080` — HTTP протокол
    - `udp://192.168.1.100:8081` — UDP протокол (быстрая отправка)
    - `tcp://192.168.1.100:8082` — TCP протокол (надежная доставка)
    - `https://192.168.1.100:8080` — HTTPS протокол (шифрование)
- `--source` — источник логов (auto, journal, eventlog, file)
- `--limit` — количество событий за цикл (по умолчанию 200)
- `--interval` — интервал отправки в секундах (по умолчанию 60)
- `--encrypt` — включить шифрование данных перед отправкой (опционально)
- `--encryption-key` — ключ шифрования (если не указан, используется ключ по умолчанию)

#### 2.2.3. UDP/TCP поддержка (client/udp_client.py)

**Класс `UDPClient`:**
- Отправка логов через UDP протокол
- Быстрая отправка без подтверждения получения
- Подходит для высокочастотных событий

**Класс `TCPClient`:**
- Отправка логов через TCP протокол
- Надежная доставка с подтверждением
- Подходит для критически важных логов

**Функция `parse_server_url()`:**
- Автоматическое определение протокола из URL
- Поддержка `udp://`, `tcp://`, `http://`, `https://`

#### 2.2.4. Шифрование данных (client/encryption.py)

**Класс `EncryptionManager`:**
- Шифрование данных перед отправкой на сервер
- Использует библиотеку `cryptography` (Fernet)
- Поддержка шифрования JSON данных

**Методы:**
- `encrypt()` / `decrypt()` — шифрование/расшифровка байтов
- `encrypt_json()` / `decrypt_json()` — шифрование/расшифровка JSON
- `get_key_string()` — получение ключа для передачи

**Безопасность:**
- Использует симметричное шифрование (Fernet)
- Ключ может быть задан пользователем или использован по умолчанию
- Рекомендуется использовать уникальные ключи для каждого агента

### 2.3. Клиент-аудитор (client/gui.py)

#### 2.3.1. Графический интерфейс

**Основное окно:**
- Двухвкладочный интерфейс (Logs, Dashboard)
- Адаптивный размер окна (1400x800)

#### 2.3.2. Вкладка "Logs" — Просмотр логов

**Фильтры:**
1. **Host** — фильтр по hostname (текстовое поле)
2. **Severity** — фильтр по уровню важности (выпадающий список: emerg, alert, crit, err, warn, notice, info, debug)
3. **Search** — поиск по содержимому сообщения (текстовое поле, поиск по Enter)
4. **Time Filter** — фильтр по времени:
   - All time — все время
   - Last hour — последний час
   - Last 6 hours — последние 6 часов
   - Last 24 hours — последние 24 часа
   - Last 7 days — последние 7 дней
   - Custom... — произвольная дата и время
5. **Limit** — количество отображаемых событий (1-1000, по умолчанию 200)

**Таблица логов:**
- Отображение колонок: ts, severity, host, unit, pid, message
- Сортировка по времени (по умолчанию — новые сверху)
- Растягивание последней колонки (message)
- Выбор строк
- **Двойной клик** — открытие окна с детальной информацией о событии

**Детальное окно события:**
- Отображение всех полей:
  - Timestamp
  - Host
  - Source
  - Severity
  - Unit
  - Process
  - PID
  - UID
  - Message
  - Raw Data (оригинальное событие)

**Кнопки:**
- **Fetch** — загрузка логов с применением фильтров
- **Dashboard** — переключение на вкладку Dashboard
- **Stats** — отображение статистики (в статусной строке)

#### 2.3.3. Вкладка "Dashboard" — Аналитика

**Графики:**

1. **Events by Severity** (круговая диаграмма):
   - Распределение событий по уровням важности
   - Цветовая индикация:
     - emerg, alert, crit, err — оттенки красного/оранжевого
     - warn — оранжевый
     - notice — золотой
     - info — синий
     - debug — серый
   - Легенда справа

2. **Events by Host (Top 10)** (столбчатая диаграмма):
   - Топ-10 хостов по количеству событий
   - Горизонтальная гистограмма
   - Подписи хостов (ограничение длины 20 символов)

3. **Events Timeline (Last 24 hours)** (линейный график):
   - Временная линия событий за последние 24 часа
   - Группировка по часам
   - Ось X — время (формат HH:mm)
   - Ось Y — количество событий

**Кнопки:**
- **Refresh Dashboard** — обновление всех графиков

#### 2.3.4. Модель данных (LogsTableModel)

- Наследование от `QAbstractTableModel`
- Динамическое обновление данных
- Поддержка стандартных методов Qt для работы с таблицами

#### 2.3.5. Клиент для связи с сервером (client/connection.py)

**Класс `ServerClient`:**
- Инкапсуляция HTTP запросов
- Методы:
  - `health()` — проверка доступности сервера
  - `fetch_logs()` — получение логов с фильтрацией
  - `get_stats()` — получение статистики
  - `send_logs()` — отправка логов (используется агентом)
- Обработка таймаутов
- Обработка ошибок HTTP

## 3. Технические характеристики

### 3.1. Производительность

- **Масштабируемость:** поддержка множества агентов (N машин)
- **Дедупликация:** предотвращение дублирования событий
- **Индексы БД:** оптимизация запросов фильтрации
- **Пакетная обработка:** массовое сохранение событий
- **Асинхронность:** FastAPI с Uvicorn для высокой нагрузки
- **Кэширование:** in-memory кэш для часто запрашиваемых данных
- **Альтернативные протоколы:** UDP для быстрой отправки, TCP для надежной доставки

### 3.2. Надежность

- **Обработка ошибок:** graceful degradation при сбоях
- **Валидация данных:** проверка корректности событий
- **Логирование:** информативные сообщения о работе системы
- **Восстановление:** автоматическая обработка прерываний

### 3.3. Безопасность (расширенная реализация для демонстрации навыков ИБ)

#### 3.3.1. Аутентификация и пароли

- **Хеширование паролей:** bcrypt с адаптивным количеством раундов (12 раундов = ~300ms)
- **Политика паролей:** минимум 8 символов, заглавные/строчные буквы, цифры, спецсимволы
- **Защита от timing attacks:** constant-time сравнение паролей через bcrypt
- **Валидация силы пароля:** проверка на распространенные слабые пароли

#### 3.3.2. Защита от атак

- **Brute force protection:** rate limiting (5 попыток за 5 минут) по IP+username
- **Блокировка аккаунтов:** автоматическая блокировка после 5 неудачных попыток на 30 минут
- **CSRF защита:** токены CSRF для всех операций изменения данных
- **SQL injection protection:** параметризованные запросы + валидация входных данных
- **XSS protection:** санитизация данных + Content Security Policy
- **Timing attacks protection:** constant-time операции для критичных проверок

#### 3.3.3. Управление сессиями

- **Безопасные токены:** генерация через `secrets.token_urlsafe(32)` (256 бит энтропии)
- **Ротация сессий:** инвалидация старых сессий при создании новой
- **HTTP-only cookies:** защита от XSS через JavaScript
- **SameSite cookies:** защита от CSRF атак
- **TTL сессий:** 24 часа с автоматической очисткой истекших

#### 3.3.4. Security Headers

- **X-Content-Type-Options:** nosniff
- **X-Frame-Options:** DENY (защита от clickjacking)
- **X-XSS-Protection:** 1; mode=block
- **Content-Security-Policy:** строгая политика безопасности контента
- **Strict-Transport-Security:** принудительное использование HTTPS
- **Referrer-Policy:** strict-origin-when-cross-origin
- **Permissions-Policy:** отключение неиспользуемых API браузера

#### 3.3.5. Валидация и санитизация

- **Валидация username:** проверка формата, длины, опасных символов
- **Санитизация поисковых запросов:** удаление SQL/XSS паттернов
- **Санитизация строк:** удаление нулевых байтов, управляющих символов
- **Ограничение длины:** защита от DoS через большие запросы

#### 3.3.6. Аудит и мониторинг

- **Журнал действий:** полное логирование всех действий пользователей
- **Отслеживание попыток входа:** счетчик неудачных попыток, время блокировки
- **IP-адреса:** сохранение IP-адресов всех действий
- **Временные метки:** точное время всех действий

#### 3.3.7. Дополнительные механизмы

- **HTTPS поддержка:** возможность запуска с SSL/TLS сертификатами
- **Шифрование данных агентами:** опциональное шифрование перед отправкой
- **Валидация входных данных:** проверка параметров запросов
- **Ограничения:** лимиты на количество записей (1-1000)

## 5. Пользовательский интерфейс

### 5.1. Назначение интерфейса

Веб-интерфейс BARSUKSIEM предназначен для аудита системных журналов удаленного персонального компьютера и предоставляет:
- мониторинг состояния агента сбора логов;
- анализ событий и их важности;
- просмотр инцидентов и деталей;
- формирование отчетов по журналам.

### 5.2. Основные экраны

1) **Удаленный аудит системных журналов (дашборд):**
   - сводные показатели по агентам, ошибкам аудита и критическим событиям;
   - лента актуальных событий и приоритетные цели аудита;
   - поиск по ПК или IP для перехода к журналам.

2) **Обозреватель журналов:**
   - фильтры по хосту, важности, времени и строке поиска;
   - таблица событий с постраничной навигацией;
   - экспорт событий в CSV.

3) **События ИБ и детали:**
   - реестр инцидентов с фильтрами и быстрым поиском;
   - карточка инцидента с хронологией, связанными логами и действиями аудитора;
   - экспорт списка инцидентов.

4) **Отчеты и аналитика:**
   - конструктор отчетов и библиотека шаблонов;
   - сводные метрики и журнал событий для включения в отчет;
   - экспорт отчета (CSV) и предварительный просмотр.

5) **Удаленные ПК:**
   - мониторинг статуса агентов и активности хостов;
   - быстрый поиск и контроль доступности.

6) **Отчеты аудита и соответствие:**
   - выбор активов и периодов комплаенс‑аудита;
   - метрики нарушений и активности пользователей;
   - экспорт данных.

### 5.2.1. Подписи к скриншотам

- Главный дашборд аудита ИБ с метриками и состоянием агентов.
- Обозреватель журналов с фильтрами и экспортом событий.
- Реестр инцидентов ИБ с фильтрацией и поиском.
- Карточка инцидента с хронологией и связанными событиями.
- Отчеты и аналитика с шаблонами и выгрузкой.
- Удаленные ПК: мониторинг статусов агентов.
- Комплаенс‑аудит: выбор активов, метрики и экспорт.

Интерфейс отражает ключевые задачи диплома: анализ системных журналов удаленного ПК, аудит событий ИБ, централизованный доступ к данным и формирование отчетности для последующего контроля соответствия требованиям безопасности.
