**Версия системы:** 0.4  
**Дата обновления:** 2024

## 🎯 Обзор системы

Система состоит из трех компонентов:
- **Сервер** - центральная машина для приема и хранения логов с веб-дашбордом (1 машина)
- **Агенты** - собирают логи на удаленных машинах (от 1 до N машин)
- **Клиент-аудитор** - веб-дашборд (доступен через браузер) или десктопный GUI (1 машина, может быть та же, что и сервер)

**Минимальная конфигурация:** 1 машина (сервер + веб-дашборд на одной машине)
**Рекомендуемая конфигурация:** 2+ машины (1 сервер + N агентов)

**Веб-дашборд** доступен через браузер по адресу `http://<IP-сервера>:8080` и включает:
- **Главная страница (Дашборд)** - `/` - карточки ключевых метрик, графики и таблица событий
- **Страница журналов** - `/logs` - полная таблица событий с расширенной фильтрацией, поиском и экспортом
- **Страница аналитики** - `/analytics` - интерактивные графики, тепловые карты и временные линии
- Автоматическое обновление данных на всех страницах
- Аутентификация и авторизация пользователей

Если Python не установлен:

### 2.4. Установите зависимости на сервере
```bash
# На Linux/macOS:
```

### 2.5. Запустите сервер
```bash
# На Linux/macOS:
python3 main.py server --host 0.0.0.0 --port 8080

# На Windows:
python main.py server --host 0.0.0.0 --port 8080

# С поддержкой HTTPS (требуются SSL сертификаты):
python3 main.py server --host 0.0.0.0 --port 8080 --ssl-cert cert.pem --ssl-key key.pem
```

**Параметры:**
- `--host 0.0.0.0` - сервер доступен со всех сетевых интерфейсов (для доступа агентов)
- `--port 8080` - порт сервера (можно изменить)
- `--db logs.db` - путь к базе данных (по умолчанию `logs.db`)
- `--ssl-cert` - путь к SSL сертификату (для HTTPS)
- `--ssl-key` - путь к приватному ключу SSL (для HTTPS)

**Ожидаемый вывод:**
```
INFO:     Started server process [12345]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8080 (Press CTRL+C to quit)
```

**Примечание:** Сервер автоматически создаст базу данных `logs.db` и пользователя по умолчанию при первом запуске.

### 2.8. Доступ к веб-дашборду

#### 2.8.1. Аутентификация
После запуска сервера веб-дашборд доступен по адресу:
- **Локально:** `http://localhost:8080` (или `https://localhost:8080` при использовании HTTPS)
- **Удаленно:** `http://<IP-сервера>:8080` (или `https://<IP-сервера>:8080`)

**При первом открытии** вы увидите страницу входа. Используйте учетные данные по умолчанию:
- **Имя пользователя:** `admin`
- **Пароль:** `Admin123!@#`

⚠️ **ВАЖНО:** 
- Пароль по умолчанию соответствует политике безопасности (8+ символов, заглавные/строчные буквы, цифры, спецсимволы)
- После первого входа рекомендуется изменить пароль администратора
- Система автоматически блокирует аккаунт после 5 неудачных попыток входа на 30 минут

**Роли пользователей:**
- **admin** - полный доступ (чтение, запись, управление пользователями, просмотр аудита)
- **auditor** - только чтение логов и статистики
- **guest** - только чтение (ограниченный доступ)

**Веб-дашборд состоит из трех основных страниц:**

**Преимущества веб-дашборда:**

- ✅ Современный и интуитивный интерфейс
- ✅ Адаптивный дизайн (работает на планшетах и телефонах)
- ✅ Автоматическое обновление данных

## 🤖 Шаг 3: Настройка АГЕНТОВ (1 или более машин)

### 3.1. Определите количество агентов
- Агент устанавливается на каждую машину, с которой нужно собирать логи
- Можно установить на любое количество машин (1, 2, 10, 100...)
- Каждая машина должна иметь доступ к серверу по сети

### 3.2. Для каждой машины-агента выполните:

#### 3.2.1. Перенесите файлы на машину агента
Скопируйте на машину агента:
- `main.py`
- `requirements-client.txt`
- Папку `client/` (со всеми файлами внутри)

**Структура на машине агента должна быть:**
```
/путь/к/проекту/
  ├── main.py
  ├── requirements-client.txt
  ├── client/
  │   ├── agent.py
  │   ├── connection.py
  │   ├── encryption.py
  │   └── udp_client.py
  └── common/              # Общие утилиты
      ├── __init__.py
      └── log_utils.py
```

#### 3.2.2. Проверьте Python на машине агента
```bash
# На Linux/macOS:
python3 --version

# На Windows:
python --version
```
**Требуется:** Python 3.10 или выше

**⚠️ Для Ubuntu 22.04:** Если pip не установлен (ошибка `No module named pip`), см. раздел 2.3 для инструкций по установке.

#### 3.2.3. Установите зависимости на машине агента
```bash
# На Linux/macOS:
cd /путь/к/проекту
python3 -m pip install -U pip
python3 -m pip install -U -r requirements-client.txt

# На Windows:
cd C:\путь\к\проекту
python -m pip install -U pip
python -m pip install -U -r requirements-client.txt

# Опционально для Windows (для лучшей производительности):
pip install pywin32
```

#### 3.2.4. Запустите агента

**На Linux-машине:**
```bash
python3 main.py agent --server http://<IP-сервера>:8080 --source journal --interval 60
```

**На Windows-машине:**
```powershell
python main.py agent --server http://<IP-сервера>:8080 --source eventlog --interval 60
```

**Параметры:**
- `--server` - URL сервера (поддерживает несколько протоколов):
  - `http://<IP-сервера>:8080` - HTTP протокол (по умолчанию)
  - `udp://<IP-сервера>:8081` - UDP протокол (быстрая отправка без подтверждения)
  - `tcp://<IP-сервера>:8082` - TCP протокол (надежная доставка)
  - `https://<IP-сервера>:8080` - HTTPS протокол (шифрованное соединение)
- `--source` - источник логов:
  - `auto` - автоматическое определение (рекомендуется)
  - `journal` - Linux journalctl
  - `eventlog` - Windows Event Log
  - `file` - файлы логов (Linux)
- `--limit 200` - количество событий за раз (по умолчанию 200)
- `--interval 60` - интервал отправки в секундах (по умолчанию 60)
- `--encrypt` - включить шифрование данных перед отправкой (опционально)
- `--encryption-key` - ключ шифрования (если не указан, используется переменная окружения ENCRYPTION_KEY или генерируется новый с предупреждением)

**⚠️ ВАЖНО по безопасности:**
- Для production **обязательно** установите переменную окружения `ENCRYPTION_KEY` или используйте параметр `--encryption-key`
- Не используйте ключ по умолчанию в production окружениях
- Ключ должен быть длиной минимум 32 символа

**Примеры использования:**

```bash
# Стандартный HTTP
python3 main.py agent --server http://192.168.1.100:8080

# UDP для быстрой отправки
python3 main.py agent --server udp://192.168.1.100:8081 --source journal

# TCP для надежной доставки
python3 main.py agent --server tcp://192.168.1.100:8082 --source journal

# HTTPS с шифрованием
python3 main.py agent --server https://192.168.1.100:8080 --encrypt

# С шифрованием данных и кастомным ключом
python3 main.py agent --server http://192.168.1.100:8080 --encrypt --encryption-key "your-secret-key"

# С шифрованием через переменную окружения (рекомендуется)
export ENCRYPTION_KEY="your-secret-key-here"
python3 main.py agent --server http://192.168.1.100:8080 --encrypt
```

**Ожидаемый вывод:**
```
Agent started. Collecting logs from: eventlog
Sending to: http://192.168.1.100:8080
Interval: 60 seconds
Press Ctrl+C to stop

[2024-01-15 10:30:00] Sent 45 logs, saved 45, skipped 0
[2024-01-15 10:31:00] Sent 12 logs, saved 12, skipped 0
```

#### 3.2.5. Особенности для Linux-агентов
Если возникают проблемы с доступом к journalctl:
```bash
# Добавьте пользователя в группу systemd-journal
sudo usermod -aG systemd-journal $USER
# Перезайдите в систему для применения изменений
```

#### 3.2.6. Особенности для Windows-агентов
- Если `pywin32` не установлен, агент использует PowerShell (работает, но медленнее)
- Для некоторых событий может потребоваться запуск от имени администратора

## 💻 Шаг 4: Настройка КЛИЕНТА-АУДИТОРА (1 машина)

> **💡 Примечание:** Начиная с версии 0.2, система включает веб-дашборд, доступный через браузер. 
> Десктопный GUI клиент (PySide6) по-прежнему доступен, но веб-дашборд рекомендуется для большинства случаев использования.

### 4.1. Выберите способ просмотра логов

**Вариант A: Веб-дашборд (рекомендуется)**
- Откройте браузер и перейдите по адресу `http://<IP-сервера>:8080`
- Не требует установки дополнительного ПО
- Работает на любой платформе
- Современный интерфейс с графиками и метриками

**Вариант B: Десктопный GUI клиент**
- Требует установки PySide6
- Работает как отдельное приложение
- Подходит для случаев, когда нужен офлайн-доступ

### 4.2. Использование веб-дашборда

Просто откройте в браузере:
```
http://<IP-сервера>:8080
```

**Навигация по страницам:**
- **Главная страница (Дашборд)** - `/` - обзор системы с метриками и графиками
- **Журналы** - `/logs` - полная таблица событий с расширенной фильтрацией
- **Аналитика** - `/analytics` - интерактивные графики и визуализации

**Использование боковой панели:**
- Кликните на иконку в левой панели для перехода на соответствующую страницу
- Активная страница подсвечивается синим цветом
- Иконка настроек открывает модальное окно с настройками автообновления

**Фильтрация и поиск:**
- На странице журналов используйте фильтры: хост, уровень важности, время, поиск по содержимому
- На странице аналитики выберите период для временного графика (24ч/7д/30д)
- Нажмите "Обновить" для ручного обновления данных
- Данные автоматически обновляются каждые 30 секунд (настраивается)

## 🔧 Решение проблем

### Проблема: "python: command not found"

**Решение:**
- На Linux/macOS используйте `python3` вместо `python`
- Убедитесь, что Python установлен и добавлен в PATH

### Проблема: "ModuleNotFoundError"

**Решение:**
1. Убедитесь, что вы установили зависимости:
   - Для сервера: `pip install -r requirements-server.txt`
   - Для агента/клиента: `pip install -r requirements-client.txt`
2. Проверьте, что используете правильную версию Python (3.10+)
3. Убедитесь, что вы находитесь в правильной директории с файлами

### Проблема: Агент не может подключиться к серверу

**Ошибка:** `Connection refused` или `Failed to establish a new connection`

**Пошаговая диагностика:**

#### Шаг 1: Проверьте, что сервер запущен
На машине сервера выполните:
```bash
# Проверьте, запущен ли сервер
ps aux | grep "main.py server"
# или
netstat -tuln | grep 8080
```

Если сервер не запущен, запустите его:
```bash
python3 main.py server --host 0.0.0.0 --port 8080
```

#### Шаг 2: Проверьте IP-адрес сервера
На машине сервера выполните:
```bash
# Узнайте IP-адрес сервера
hostname -I
# или
ip addr show | grep "inet "
```

**⚠️ ВАЖНО для VirtualBox:**
- Если обе VM используют **NAT** - они **не видят друг друга**!
- Используйте **Bridged Adapter** или **Internal Network** для связи между VM
- IP вида `10.0.2.x` - это NAT, VM не могут общаться напрямую

#### Шаг 3: Проверьте доступность сервера с машины агента
На машине агента выполните:
```bash
# Проверьте доступность сервера
curl http://<IP-сервера>:8080/health

# Если curl не установлен:
sudo apt install curl
```

**Ожидаемый ответ:** `{"status":"ok"}`

Если не работает:
- Проверьте, что IP правильный
- Проверьте настройки сети в VirtualBox

#### Шаг 4: Настройка сети в VirtualBox (если сервер и агент на разных VM)

**Вариант A: Bridged Adapter (рекомендуется)**
1. В VirtualBox выберите VM → Настройки → Сеть
2. Адаптер 1 → Тип подключения: **Сетевой мост (Bridged Adapter)**
3. Примените для обеих VM (сервер и агент)
4. Перезапустите VM
5. Узнайте новые IP-адреса: `hostname -I` на каждой VM
6. Используйте новый IP сервера в команде агента

**Вариант B: Internal Network**
1. В VirtualBox выберите VM → Настройки → Сеть
2. Адаптер 1 → Тип подключения: **Внутренняя сеть (Internal Network)**
3. Имя сети: `audit-network` (одинаковое для обеих VM)
4. Перезапустите VM
5. Узнайте IP-адреса: `hostname -I`
6. Используйте IP сервера в команде агента

#### Шаг 5: Проверьте файрвол
На машине сервера:
```bash
# Ubuntu/Debian
sudo ufw status
# Если файрвол активен, откройте порт:
sudo ufw allow 8080/tcp
```

#### Шаг 6: Убедитесь, что сервер запущен с правильными параметрами
Сервер должен быть запущен с `--host 0.0.0.0`:
```bash
python3 main.py server --host 0.0.0.0 --port 8080
```

**Неправильно:** `--host 127.0.0.1` (доступен только локально)

#### Шаг 7: Если сервер и агент на одной машине
Используйте `127.0.0.1` или `localhost`:
```bash
python3 main.py agent --server http://127.0.0.1:8080 --source journal --interval 60
```

## 🔒 Анализ инцидентов информационной безопасности

Система автоматически анализирует события и выявляет инциденты ИБ с помощью rule-based анализа.

### Реализованные правила:

1. **R001 - Brute Force Detection**
   - Обнаруживает 5+ неудачных попыток входа за 10 минут
   - Критичность: HIGH

2. **R002 - Unauthorized Access (Night Admin Login)**
   - Обнаруживает вход администратора в нерабочее время (00:00-06:00)
   - Критичность: MEDIUM

3. **R003 - Suspicious Activity (PowerShell Download)**
   - Обнаруживает запуск PowerShell с одновременной загрузкой данных
   - Критичность: HIGH

4. **R004 - Log Tampering Detection**
   - Обнаруживает попытки удаления или изменения логов
   - Критичность: CRITICAL

5. **R005 - Privilege Escalation Detection**
   - Корреляция событий: неудачные входы → успешный вход → вход с правами
   - Критичность: CRITICAL

### API для работы с инцидентами:

- `GET /api/incidents` - получение списка инцидентов с фильтрацией

## 🎯 API эндпоинты сервера

Для разработчиков:

### Публичные эндпоинты (не требуют аутентификации):

- `GET /health` - проверка доступности сервера
- `POST /logs` - отправка логов от агентов (используется агентами)

### Эндпоинты аутентификации:

- `POST /api/auth/login` - вход в систему
  - Тело запроса: `{"username": "admin", "password": "Admin123!@#"}`
  - Возвращает токен сессии и устанавливает cookie
  - **Защита:** Rate limiting (5 попыток за 5 минут), блокировка аккаунта после 5 неудачных попыток
- `POST /api/auth/logout` - выход из системы (требует аутентификации и CSRF токен)
- `GET /api/auth/me` - получение информации о текущем пользователе (требует аутентификации)

### Защищенные эндпоинты (требуют аутентификации):

- `GET /` - главная страница веб-дашборда (HTML) - перенаправляет на `/login` если не авторизован
- `GET /logs` - страница журналов событий (HTML) - перенаправляет на `/login` если не авторизован
- `GET /analytics` - страница аналитики (HTML) - перенаправляет на `/login` если не авторизован
- `GET /login` - страница входа (HTML)
- `GET /logs?host=...&severity=...&since=...&search=...&limit=...&offset=...` - получение логов с фильтрацией
  - `host` - фильтр по хосту (санитизируется перед использованием)
  - `severity` - фильтр по уровню важности (err, warn, info, debug и т.д.)
  - `since` - фильтр по времени (ISO формат, например: "2024-01-15T10:00:00")
  - `search` - поиск по содержимому сообщения (LIKE поиск, санитизируется)
  - `limit` - максимальное количество событий (1-1000)
  - `offset` - смещение для пагинации
  - **Кэширование:** результаты кэшируются на 60 секунд (проверка кэша происходит ПЕРЕД запросом к БД)
  - **Безопасность:** Все параметры санитизируются перед использованием в запросах
- `GET /stats` - статистика по логам
  - **Кэширование:** результаты кэшируются на 5 минут
- `GET /api/audit?limit=...&offset=...` - журнал аудита действий пользователей (только для админов)
  - Показывает все действия пользователей: входы, выходы, просмотры логов и т.д.
  - Требует права `manage_users` (обычно только для роли admin)
- `GET /api/incidents` - получение инцидентов ИБ с фильтрацией
  - Параметры: `incident_type`, `severity`, `status`, `since`, `limit`, `offset`
  - Автоматически выявляются при сохранении событий
- `GET /api/incidents/rules` - информация о правилах выявления инцидентов
- `GET /api/incidents/stats` - статистика по инцидентам ИБ
- `GET /static/*` - статические файлы (CSS, JS) для веб-дашборда

### Аутентификация через API:

Все защищенные эндпоинты требуют наличия cookie `session_token`, который устанавливается при успешном входе через `/api/auth/login`.

**Операции изменения данных** (POST, PUT, DELETE) также требуют CSRF токен в заголовке `X-CSRF-Token`.

**Пример использования с curl:**
```bash
# Вход
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"Admin123!@#"}' \
  -c cookies.txt

# Получение CSRF токена из ответа (сохраните в переменную)
CSRF_TOKEN=$(curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"Admin123!@#"}' \
  -c cookies.txt -b cookies.txt | grep -o '"csrf_token":"[^"]*' | cut -d'"' -f4)

# Получение логов (с использованием cookie)
curl http://localhost:8080/logs?limit=10 -b cookies.txt

# Получение статистики
curl http://localhost:8080/stats -b cookies.txt

# Выход (требует CSRF токен)
curl -X POST http://localhost:8080/api/auth/logout \
  -b cookies.txt \
  -H "X-CSRF-Token: $CSRF_TOKEN"
```

**Примеры запросов:**

```bash
# Проверка здоровья
curl http://localhost:8080/health

# Получение статистики (требует аутентификации)
curl http://localhost:8080/stats -b cookies.txt

# Получение логов с фильтром по severity
curl "http://localhost:8080/logs?limit=10&severity=err" -b cookies.txt

# Поиск по содержимому
curl "http://localhost:8080/logs?search=error&limit=50" -b cookies.txt

# Фильтр по времени (последний час)
curl "http://localhost:8080/logs?since=2024-01-15T10:00:00" -b cookies.txt

# Комбинированные фильтры
curl "http://localhost:8080/logs?host=server1&severity=err&search=failed&limit=100" -b cookies.txt
```
