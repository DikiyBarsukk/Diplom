# Безопасность BARSUKSIEM

## Обзор

В проекте реализован набор механизмов, достаточных для дипломной демонстрации безопасной клиент-серверной системы аудита.

Ключевые блоки:

- bcrypt-хеширование паролей
- политика сложности пароля
- session cookie аутентификация
- CSRF-защита через `X-CSRF-Token`
- rate limiting для login и ingest
- sanitization входных данных
- аудит действий пользователей
- security headers

## Пароли и пользователи

- пароли хранятся в виде bcrypt hash
- проверка выполняется через constant-time механизмы bcrypt
- применяется политика минимальной сложности:
  - минимум 8 символов
  - заглавные и строчные буквы
  - цифры
  - специальные символы
- простой hardcoded admin `admin/Admin123!@#` удален
- bootstrap admin создается только на пустой БД

## Bootstrap admin

Безопасный сценарий запуска:

- задать `BARSUKSIEM_BOOTSTRAP_ADMIN_USERNAME`
- задать `BARSUKSIEM_BOOTSTRAP_ADMIN_PASSWORD`
- если пароль не задан, сервер генерирует его один раз и печатает в startup log

Это безопаснее, чем хранить постоянные дефолтные credentials в коде и документации.

## Session и CSRF flow

- вход выполняется через `POST /api/auth/login`
- сервер создает `session_token` cookie
- `GET /api/auth/me` возвращает текущего пользователя и обновляет CSRF token через header `X-CSRF-Token`
- защищенные методы изменения состояния используют `X-CSRF-Token`
- `POST /api/auth/logout` без CSRF token возвращает `403`

## Rate limiting

Реализованы два основных ограничения:

- login rate limit для защиты от brute force
- ingest rate limit для защиты API приема логов

Это снижает риск перебора паролей и перегрузки публичного ingest endpoint.

## Security headers

Приложение добавляет стандартные browser-side защиты через middleware.

Типовые заголовки:

- `X-Content-Type-Options`
- `X-Frame-Options`
- `X-XSS-Protection`
- `Content-Security-Policy`
- `Strict-Transport-Security`
- `Referrer-Policy`
- `Permissions-Policy`

## Санитизация данных

Используется `InputSanitizer`:

- ограничение длины строк
- очистка управляющих символов
- безопасная обработка search query
- валидация username

Это снижает риски SQL injection, XSS и ошибок обработки пользовательских данных.

## Аудит действий

В `audit_log` фиксируются:

- login success / login failed
- logout
- просмотры защищенных разделов и статистики
- обращения к журналу аудита

Журнал действий может использоваться как для расследования, так и для демонстрации accountability на защите.

## Конфигурация безопасности через env

- `BARSUKSIEM_COOKIE_SECURE`
- `BARSUKSIEM_COOKIE_SAMESITE`
- `BARSUKSIEM_SESSION_MAX_AGE`
- `CORS_ALLOWED_ORIGINS`
- `BARSUKSIEM_BOOTSTRAP_ADMIN_USERNAME`
- `BARSUKSIEM_BOOTSTRAP_ADMIN_PASSWORD`
- `BARSUKSIEM_DEMO_MODE`

## Что проверено тестами

Тестами подтверждены:

- успешный login
- доступ к `GET /api/auth/me`
- отказ logout без CSRF token
- корректный logout с валидным CSRF token
- доступ к защищенным API только после аутентификации

## Рекомендации для production

- включить HTTPS и `BARSUKSIEM_COOKIE_SECURE=true`
- ограничить `CORS_ALLOWED_ORIGINS` только доверенными origin
- вынести SQLite на более производительное хранилище при росте нагрузки
- добавить централизованный мониторинг и rotation audit log
- использовать управляемую выдачу bootstrap credentials при первом старте
