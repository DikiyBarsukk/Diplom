# 🔒 Документация по механизмам информационной безопасности

## Обзор

Система реализует комплексный подход к информационной безопасности, демонстрируя навыки инженера ИБ. Все механизмы безопасности разработаны с учетом современных стандартов и best practices.

---

## 1. Аутентификация и управление паролями

### 1.1. Хеширование паролей (bcrypt)

**Реализация:** `server/security.py` → `PasswordSecurity`

**Механизм:**
- Использование алгоритма **bcrypt** вместо SHA-256
- Автоматическая генерация соли для каждого пароля
- Настраиваемое количество раундов (по умолчанию 12 = ~300ms на современном CPU)
- Защита от rainbow table атак через уникальные соли

**Код:**
```python
password_hash = PasswordSecurity.hash_password(password, rounds=12)
is_valid = PasswordSecurity.verify_password(password, password_hash)
```

**Преимущества:**
- ✅ Адаптивное хеширование (можно увеличить сложность со временем)
- ✅ Защита от timing attacks через constant-time сравнение
- ✅ Устойчивость к brute force атакам

### 1.2. Политика паролей

**Требования к паролю:**
- Минимум 8 символов
- Хотя бы одна заглавная буква
- Хотя бы одна строчная буква
- Хотя бы одна цифра
- Хотя бы один специальный символ
- Проверка на распространенные слабые пароли

**Реализация:** `PasswordSecurity.validate_password_strength()`

## 2. Защита от атак

### 2.1. Защита от Brute Force атак

**Реализация:** `server/security.py` → `RateLimiter`

**Механизм:**
- Rate limiting по комбинации IP-адреса и username
- Максимум 5 неудачных попыток за 5 минут
- Автоматическая блокировка аккаунта после 5 неудачных попыток на 30 минут
- Автоматическая очистка старых записей

**Особенности:**
- Отдельный счетчик неудачных попыток для каждого пользователя
- Защита от распределенных атак через комбинированный ключ (IP + username)
- Информативные сообщения об ошибках без раскрытия информации о существовании пользователя

**Код:**
```python
is_allowed, retry_after = rate_limiter.check_rate_limit(
    key=f"{ip}:{username}",
    max_attempts=5,
    window_seconds=300
)
```

### 2.2. Защита от Timing Attacks

**Реализация:** `PasswordSecurity.verify_password()`

**Механизм:**
- Использование `bcrypt.checkpw()` для constant-time сравнения
- При ошибке выполнения - фиктивная операция хеширования для сохранения времени выполнения
- Защита от утечки информации через время ответа

**Код:**
```python
try:
    return bcrypt.checkpw(password_bytes, hash_bytes)
except Exception:
    # Constant-time операция для предотвращения timing attacks
    bcrypt.hashpw(b"dummy", bcrypt.gensalt())
    return False
```

### 2.3. Защита от CSRF атак

**Реализация:** `server/security.py` → `CSRFProtection`

**Механизм:**
- Генерация уникального CSRF токена для каждой сессии
- Хранение токена в БД, связанного с сессией
- Проверка токена через `secrets.compare_digest()` (constant-time сравнение)
- Требование CSRF токена для всех операций изменения данных (POST, PUT, DELETE)

**Использование:**
- CSRF токен генерируется при создании сессии
- Токен передается клиенту в заголовке `X-CSRF-Token`
- Клиент отправляет токен в заголовке `X-CSRF-Token` при операциях изменения данных

**Код:**
```python
csrf_token = CSRFProtection.generate_token()
is_valid = CSRFProtection.validate_token(request_token, stored_token)
```

### 2.4. Защита от SQL Injection

**Механизм:**
- Использование параметризованных запросов во всех SQL операциях
- Валидация и санитизация входных данных перед использованием в запросах
- Проверка на опасные SQL конструкции в пользовательском вводе

**Реализация:**
- Все SQL запросы используют `?` placeholders
- Валидация через `InputSanitizer.validate_username()`
- Паттерны SQL injection блокируются на уровне валидации

### 2.5. Защита от XSS

**Реализация:** `server/security.py` → `InputSanitizer`

**Механизм:**
- Санитизация всех пользовательских данных перед отображением
- Проверка на опасные HTML/JavaScript конструкции
- Content Security Policy (CSP) заголовки
- Экранирование специальных символов

**Паттерны блокировки:**
- `<script>` теги
- `javascript:` протокол
- Event handlers (`onclick`, `onerror`, etc.)
- `<iframe>`, `<object>`, `<embed>` теги

## 3. Управление сессиями

### 3.1. Безопасные токены сессий

**Механизм:**
- Генерация токенов через `secrets.token_urlsafe(32)` (256 бит энтропии)
- Хранение токенов в БД с привязкой к пользователю
- TTL сессий: 24 часа
- Автоматическая очистка истекших сессий

### 3.2. Ротация сессий

**Механизм:**
- При создании новой сессии все старые сессии пользователя инвалидируются
- Защита от session fixation атак
- Отслеживание времени последней активности

### 3.3. Безопасные Cookies

**Настройки:**
- `HttpOnly=True` - защита от XSS через JavaScript
- `SameSite=Lax` - защита от CSRF
- `Secure=False` (в production должно быть `True` для HTTPS)
- `max_age=86400` (24 часа)

### 3.4. CSRF токены в сессиях

**Механизм:**
- Каждая сессия имеет уникальный CSRF токен
- Токен хранится в БД и связан с сессией
- Токен обновляется при создании новой сессии

## 4. Security Headers

**Реализация:** `server/security.py` → `SecurityHeaders`

**Устанавливаемые заголовки:**

### 4.1. X-Content-Type-Options: nosniff
- Предотвращает MIME type sniffing
- Защита от атак через подмену типа контента

### 4.2. X-Frame-Options: DENY
- Защита от clickjacking атак
- Запрещает встраивание страницы в iframe

### 4.3. X-XSS-Protection: 1; mode=block
- Включает встроенную защиту браузера от XSS
- Блокирует отображение страницы при обнаружении XSS

### 4.4. Content-Security-Policy
- Строгая политика безопасности контента
- Разрешает только необходимые источники для скриптов, стилей и изображений
- Запрещает inline скрипты (кроме необходимых для работы)

### 4.5. Strict-Transport-Security
- Принудительное использование HTTPS
- `max-age=31536000` (1 год)
- `includeSubDomains` - применяется ко всем поддоменам

### 4.6. Referrer-Policy: strict-origin-when-cross-origin
- Контроль передачи referrer информации
- Защита от утечки информации через referrer

### 4.7. Permissions-Policy
- Отключает неиспользуемые API браузера
- Защита от утечки информации через геолокацию, камеру, микрофон и т.д.

## 5. Валидация и санитизация данных

**Реализация:** `server/security.py` → `InputSanitizer`

### 5.1. Валидация имени пользователя

**Требования:**
- Минимум 3 символа, максимум 50 символов
- Только буквы, цифры, дефисы и подчеркивания
- Проверка на SQL injection паттерны
- Проверка на XSS паттерны

### 5.2. Санитизация поисковых запросов

**Механизм:**
- Удаление опасных SQL конструкций
- Ограничение длины до 500 символов
- Удаление нулевых байтов
- Удаление управляющих символов

### 5.3. Санитизация строк

**Механизм:**
- Удаление нулевых байтов (`\x00`)
- Удаление управляющих символов (кроме `\n` и `\t`)
- Ограничение длины
- Trim пробелов

## 6. Аудит безопасности

### 6.1. Журнал действий пользователей

**Таблица:** `audit_log`

**Записывается:**
- Все попытки входа (успешные и неудачные)
- Выходы из системы
- Просмотры логов и статистики
- IP-адреса всех действий
- Временные метки
- Детали действий

### 6.2. Отслеживание неудачных попыток

**Таблица:** `users`

**Поля:**
- `failed_login_attempts` - счетчик неудачных попыток
- `locked_until` - время разблокировки аккаунта
- `last_login` - время последнего успешного входа
- `password_changed_at` - время последней смены пароля

## 7. Архитектурные решения безопасности

### 7.1. Разделение ответственности

- Отдельный модуль `server/security.py` для всех механизмов безопасности
- Четкое разделение между аутентификацией и авторизацией
- Изоляция критичных операций

### 7.2. Defense in Depth

- Множественные уровни защиты
- Защита на уровне приложения, БД и сети
- Валидация на каждом этапе обработки данных

### 7.3. Fail Secure

- При ошибках безопасности - блокировка доступа
- Логирование всех подозрительных действий
- Информирование пользователя без раскрытия деталей

## 8. Рекомендации для production

### 8.1. HTTPS

- Обязательное использование HTTPS в production
- Установить `Secure=True` в cookies
- Использовать валидные SSL сертификаты

### 8.2. CORS

- Ограничить `allow_origins` конкретными доменами
- Не использовать `allow_origins=["*"]` в production

### 8.3. Rate Limiting

- Рассмотреть использование Redis для распределенного rate limiting
- Настроить более строгие лимиты для критичных операций

### 8.4. Мониторинг

- Настроить алерты на множественные неудачные попытки входа
- Мониторинг журнала аудита
- Отслеживание подозрительной активности

### 8.5. Пароли

- Требовать смену пароля по умолчанию при первом входе
- Реализовать политику истечения паролей
- Рассмотреть двухфакторную аутентификацию (2FA)
