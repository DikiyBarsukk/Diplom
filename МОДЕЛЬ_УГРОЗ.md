# Модель угроз BARSUKSIEM

## Контекст системы

Система принимает логи с удаленных машин, хранит их, анализирует на наличие инцидентов и предоставляет web/API доступ аудитору и администратору.

Ключевой поток данных:

`агент -> POST /api/logs -> normalize -> storage -> incident rules -> web/API clients`

## Активы

- журналы событий и их содержимое
- информация об инцидентах
- учетные записи пользователей и роли
- session cookie и CSRF token
- audit log действий пользователей
- bootstrap admin credentials

## Основные угрозы

### 1. Подбор пароля и brute force login

Риск:

- подбор пароля к административной учетной записи

Контрмеры:

- rate limiting login
- временная блокировка учетной записи при повторных неудачных попытках
- политика сложности паролей
- удаление hardcoded admin credentials

### 2. CSRF на изменяющих запросах

Риск:

- принудительный logout или другое действие от имени авторизованного пользователя

Контрмеры:

- отдельный `X-CSRF-Token`
- валидация CSRF token на защищенных POST-запросах
- получение актуального токена через `/api/auth/me`

### 3. Компрометация session cookie

Риск:

- несанкционированный доступ к защищенному API

Контрмеры:

- `HttpOnly` cookie
- configurable `SameSite`
- configurable `secure` flag
- ограниченный lifetime session

### 4. Вредоносные входные данные

Риск:

- XSS, инъекции, ошибки обработки search query

Контрмеры:

- `InputSanitizer`
- ограничение длины входных значений
- безопасный рендеринг в UI helper-слое
- security headers

### 5. Перегрузка ingest endpoint

Риск:

- flood на публичный маршрут приема логов

Контрмеры:

- rate limiting ingest
- базовая валидация входного payload
- отказ на пустых и некорректных запросах

### 6. Несанкционированный доступ к журналам и статистике

Риск:

- чтение событий и инцидентов без аутентификации

Контрмеры:

- session-based auth для `/api/logs`, `/api/stats`, `/api/incidents`, `/api/audit`
- role/permission checks для административных функций

## Доверительные границы

- агент и сервер взаимодействуют по сети
- браузер и сервер взаимодействуют через cookie session
- desktop-клиент работает через login в защищенный API
- SQLite хранит критичные данные на стороне сервера и должен рассматриваться как чувствительный актив

## Остаточные риски

- SQLite подходит для дипломной версии, но ограничен для многопользовательской промышленной эксплуатации
- bootstrap password в startup log допустим для demo, но требует отдельной operational-процедуры в production
- rule-based анализ не покрывает сложные поведенческие аномалии без расширения набора правил

## Приоритеты дальнейшего усиления

1. HTTPS по умолчанию
2. ротация секретов и bootstrap flow без логирования пароля
3. перенос хранилища на PostgreSQL
4. вынос ingest в очередь и отдельный worker
5. расширение мониторинга и alerting
