# 🛡️ Модель угроз и нарушителя системы аудита логов

Данный документ описывает модель угроз и нарушителя для системы аудита логов, а также механизмы противодействия, реализованные в системе.

## 3. Механизмы противодействия угрозам

Система реализует комплексный подход к защите от описанных угроз.

### 3.1. Противодействие несанкционированному доступу

#### Таблица реализованных механизмов

| Механизм | Описание | Параметры | Правило обнаружения |
|---|---|---|---|
| **Аутентификация** | Система входа с логином и паролем | • Роли: admin, auditor, guest<br>• Сессии: TTL 24 часа<br>• Проверка прав на каждом эндпоинте | R002 |
| **Rate Limiting** | Ограничение попыток входа | • Максимум 5 попыток за 5 минут<br>• Блокировка на 30 минут<br>• Отслеживание по IP + username | R001 |
| **CSRF Защита** | Защита от CSRF атак | • Токены для операций изменения<br>• Constant-time сравнение<br>• Связь с сессией | - |
| **Валидация данных** | Проверка входных данных | • Санитизация всех параметров<br>• Проверка на SQL injection<br>• Проверка на XSS | - |

**Сценарий работы:**
```
Угроза: Внешний злоумышленник пытается подобрать пароль
→ Система фиксирует множественные неудачные попытки входа
→ Правило R001 обнаруживает паттерн brute force (5+ попыток за 10 минут)
→ Аккаунт блокируется автоматически на 30 минут
→ Инцидент регистрируется в системе с критичностью HIGH
→ Администратор получает уведомление через логирование
```

#### Таблица реализованных механизмов

| **HTTPS** | Шифрование передачи | • SSL/TLS сертификаты<br>• Опциональное шифрование агентов<br>• Безопасные cookies | - |
| **Мониторинг** | Отслеживание активности | • Аудит всех входов<br>• Отслеживание IP-адресов<br>• Временные метки | R002 |

**Сценарий работы:**
```
Угроза: Компрометация учетных данных администратора
→ Злоумышленник использует украденные учетные данные
→ Вход происходит в нерабочее время (02:30)
→ Правило R002 обнаруживает аномалию (admin login в 00:00-06:00)
→ Инцидент регистрируется с критичностью MEDIUM
→ Администратор получает уведомление через логирование
→ Возможность немедленной блокировки аккаунта
```

### 3.3. Противодействие подмене логов

#### Таблица реализованных механизмов

| Механизм | Описание | Параметры | Правило обнаружения |
|---|---|---|---|
| **Обнаружение команд** | Мониторинг попыток удаления | • Ключевые слова: journalctl --vacuum, wevtutil cl, rm /var/log<br>• Критичность: CRITICAL | R004 |
| **Дедупликация** | Защита целостности | • SHA-1 hash для каждого события<br>• Невозможность незаметного изменения<br>• Хранение raw данных | - |
| **Централизованное хранение** | Независимое хранилище | • Логи на центральном сервере<br>• Независимо от локальных журналов<br>• Временные метки | - |
| **Аудит действий** | Логирование операций | • Все действия пользователей<br>• IP-адреса и временные метки<br>• Невозможность удаления без прав | - |

**Сценарий работы:**
```
Угроза: Внутренний пользователь пытается удалить логи
→ Пользователь выполняет команду: journalctl --vacuum-time=1d
→ Система фиксирует событие с ключевым словом "vacuum"
→ Правило R004 обнаруживает попытку удаления логов
→ Инцидент регистрируется с критичностью CRITICAL
→ Администратор получает немедленное уведомление
→ Действие логируется в журнале аудита
→ Логи уже сохранены в центральном хранилище
```


## 5. Дополнительные меры защиты

### 5.1. Security Headers

Система автоматически устанавливает следующие заголовки безопасности:

- `X-Content-Type-Options: nosniff` - защита от MIME type sniffing
- `X-Frame-Options: DENY` - защита от clickjacking
- `X-XSS-Protection: 1; mode=block` - защита от XSS
- `Content-Security-Policy` - строгая политика безопасности контента
- `Strict-Transport-Security` - принудительное использование HTTPS
- `Referrer-Policy` - контроль передачи referrer информации

### 5.2. Защита от SQL Injection

- ✅ Параметризованные SQL запросы
- ✅ Валидация и санитизация входных данных
- ✅ Проверка на опасные SQL конструкции

### 5.3. Защита от XSS

- ✅ Санитизация всех пользовательских данных
- ✅ Content Security Policy
- ✅ Экранирование специальных символов

### 5.4. Защита от CSRF

- ✅ CSRF токены для всех операций изменения данных
- ✅ Проверка через `secrets.compare_digest()` (constant-time)
- ✅ Токены связаны с сессией

## 6. Рекомендации по снижению рисков

### 6.1. Для снижения риска несанкционированного доступа:

1. ✅ Использовать HTTPS в production
2. ✅ Регулярно менять пароли
3. ✅ Использовать двухфакторную аутентификацию (2FA) - рекомендуется для будущих версий
4. ✅ Ограничить доступ к серверу на уровне файрвола
5. ✅ Регулярно проверять журнал аудита

### 6.2. Для снижения риска компрометации учётных данных:

1. ✅ Использовать сильные пароли (система проверяет автоматически)
2. ✅ Не использовать один пароль для нескольких систем
3. ✅ Регулярно менять пароли
4. ✅ Использовать HTTPS для защиты передачи данных
5. ✅ Мониторить подозрительную активность (входы в нерабочее время)

### 6.3. Для снижения риска подмены логов:

3. ✅ Ограничение прав доступа к серверам
4. ✅ Использование read-only доступов где возможно

### 6.4. Для снижения риска удаления следов атаки:

1. ✅ Централизованное хранение логов

4. ✅ Аудит всех действий пользователей
5. ✅ Немедленное реагирование на обнаруженные инциденты

## 7. Метрики эффективности защиты

### 7.1. Обнаружение инцидентов

Система автоматически обнаруживает следующие типы инцидентов:

- **Brute Force атаки:** Обнаруживаются в реальном времени при достижении порога (5 попыток за 10 минут)
- **Подозрительные входы:** Обнаруживаются при входе администратора в нерабочее время
- **Подозрительная активность:** Обнаруживается при корреляции PowerShell и загрузки данных
- **Попытки удаления логов:** Обнаруживаются при выполнении команд удаления/очистки
- **Повышение привилегий:** Обнаруживается через корреляцию последовательности событий

### 7.2. Время обнаружения

- **Brute Force:** < 10 минут (временное окно правила)
- **Подозрительные входы:** Мгновенно (при входе)
- **Подозрительная активность:** < 5 минут (временное окно корреляции)
- **Удаление логов:** Мгновенно (при выполнении команды)
- **Повышение привилегий:** < 30 минут (временное окно корреляции)

### 7.3. Ложные срабатывания

Система спроектирована для минимизации ложных срабатываний:

- Пороги настроены на основе best practices
- Временные окна учитывают нормальную активность
- Корреляция требует последовательности событий
