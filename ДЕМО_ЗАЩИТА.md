# Памятка для защиты BARSUKSIEM

## Старт перед комиссией

На серверной Ubuntu ВМ:

```bash
cd /opt/barsuksiem/server-pack
./reset-demo.sh
```

После этого стенд уже готов к показу.

Вход:
- `admin / admin123`
- Ubuntu: `http://127.0.0.1:8080/login`
- дополнительный доступ с Windows-хоста: `http://192.168.0.159:8080/login`

## Сценарии и хосты

| Правило | Хост | Что говорить |
|---|---|---|
| `R001` brute force | `bf-srv-01` | Система видит серию неудачных входов за короткое окно и автоматически поднимает инцидент высокой важности. |
| `R002` night admin login | `night-admin-01` | Выявляется аномальный вход администратора в ночное время, даже если это единичное событие. |
| `R003` PowerShell download | `powershell-01` | Корреляция PowerShell и сетевой загрузки даёт сигнал о подозрительной активности. |
| `R004` log tampering | `tamper-01` | Попытка очистки или удаления журналов трактуется как сокрытие следов и помечается критически. |
| `R005` privilege escalation | `priv-esc-01` | Цепочка failed login -> success -> privileged login показывает попытку повышения привилегий. |

## Что открывать на экранах

### 1. Dashboard `/`

Что говорить:
- система получает телеметрию и агрегирует её в единой панели;
- оператор сразу видит риск, открытые инциденты и свежесть потока;
- это стартовая точка для triage и расследования.

### 2. Incidents `/incidents`

Что говорить:
- rules engine выявляет 5 типов инцидентов;
- классификация идёт по типу, severity и связанным событиям;
- список уже готов для фильтрации и расследования.

Фильтры, которые удобно использовать:
- по `severity`: `critical`, `high`, `medium`;
- по поиску: `bf-srv-01`, `night-admin-01`, `powershell-01`, `tamper-01`, `priv-esc-01`.

### 3. Incident details `/incidents/details?id=...`

Что говорить:
- карточка инцидента показывает, почему правило сработало;
- внутри есть контекст, статус и переход к связанным логам;
- это уже рабочий объект расследования, а не просто запись в журнале.

### 4. Logs `/logs`

Что говорить:
- оператор может перейти от карточки инцидента к исходным событиям;
- доступны фильтры по хосту, severity и строке поиска;
- это связывает сырые события с rule-based аналитикой.

Поисковые запросы по хостам:
- `bf-srv-01`
- `night-admin-01`
- `powershell-01`
- `tamper-01`
- `priv-esc-01`

### 5. Desktop-клиент

Что говорить:
- те же данные доступны аудитору и вне браузера;
- desktop — резервная витрина для демонстрации и дополнительный интерфейс контроля.

## Канонический порядок показа

1. Войти в web UI.
2. Показать dashboard как общую картину состояния.
3. Перейти в список инцидентов.
4. Открыть 2-3 карточки расследования.
5. Показать исходные логи по одному кейсу.
6. При необходимости кратко открыть desktop-клиент.

## Короткий narrative

- Система собирает и нормализует события из разных источников.
- События сохраняются в хранилище.
- Rule-based движок выявляет инциденты и формирует карточки расследования.
- Оператор получает единый интерфейс для triage, анализа и демонстрации состояния стенда.
