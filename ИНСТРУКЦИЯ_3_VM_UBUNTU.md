# BARSUKSIEM: простая установка на 3 Ubuntu ВМ

Это главный пошаговый гайд для дипломной версии BARSUKSIEM.

Главная идея здесь простая:
- ничего не раскладываем вручную по файлам;
- не создаем обязательные `env`-переменные;
- не ищем пароль в startup log;
- просто копируем готовые папки из `deploy/`, ставим зависимости и запускаем скрипты.

После этой инструкции у вас будет такая схема:
- Ubuntu ВМ 1: сервер BARSUKSIEM
- Ubuntu ВМ 2: агент сбора логов
- Ubuntu ВМ 3: рабочее место аудитора

## 1. Что нужно заранее

Подготовьте:
- 3 виртуальные машины Ubuntu
- сеть между ними
- доступ к терминалу на всех трех ВМ
- файл проекта BARSUKSIEM на вашей основной машине

Инструкция ориентирована на:
- Ubuntu 22.04 LTS
- Ubuntu 24.04 LTS

Для примера будем использовать такие адреса:
- сервер: `192.168.0.159`
- агент: `192.168.0.119`
- аудитор: `192.168.0.160`

Если у вас другие IP-адреса, система все равно будет работать. Тогда позже нужно будет заменить адрес сервера `192.168.0.159` в двух файлах:
- `run-agent.sh`
- `run-auditor-gui.sh`

## 2. Что уже подготовлено в проекте

В проекте уже есть 3 готовые папки для развертывания:
- `deploy/server-pack`
- `deploy/agent-pack`
- `deploy/auditor-pack`

Именно их нужно копировать на виртуальные машины.

Важно:
- server-pack содержит только актуальный frontend-контур;
- исторический UI-слой не включается в deploy;
- канонический active frontend описан в [web/README.md](/E:/МТУСИ%202025-2026/Диплом/ПО/web/README.md).

## 3. Логин и пароль по умолчанию

## 4. Куда копировать папки

Рекомендуемые каталоги на Ubuntu:
- сервер: `/opt/barsuksiem/server-pack`
- агент: `/opt/barsuksiem/agent-pack`
- аудитор: `/opt/barsuksiem/auditor-pack`

Если хотите, можете выбрать другие пути. Но в этой инструкции используются именно эти каталоги.

## 5. Как копировать готовые папки на ВМ

На вашей основной машине откройте терминал в корне проекта и выполните:

```bash
scp -r deploy/server-pack user@192.168.0.159:/opt/barsuksiem/
scp -r deploy/agent-pack user@192.168.0.119:/opt/barsuksiem/
scp -r deploy/auditor-pack user@192.168.0.160:/opt/barsuksiem/
```

Если `scp` нет, можно использовать:
- общую папку виртуальной машины;
- архив `.zip` или `.tar.gz`;
- SFTP;
- обычное копирование через файловый менеджер.

Главное правило:
- на сервер копируйте только `server-pack`;
- на агент только `agent-pack`;
- на аудитора только `auditor-pack`.

## 6. Настройка Ubuntu ВМ 1: сервер

### 6.1 Создайте базовый каталог

```bash
sudo mkdir -p /opt/barsuksiem
sudo chown -R $USER:$USER /opt/barsuksiem
```

Если папка уже есть, просто переходите дальше.

### 6.2 Перейдите в каталог сервера

```bash
cd /opt/barsuksiem/server-pack
```

### 6.3 Сделайте скрипты исполняемыми

```bash
chmod +x *.sh
```

### 6.4 Установите зависимости

```bash
./install.sh
```

Что делает этот скрипт:
- ставит `python3`, `python3-venv`, `python3-pip`, `curl` через `apt`;
- создает `.venv`;
- устанавливает Python-зависимости из `requirements-server.txt`.

### 6.5 Запустите сервер

```bash
./run-server.sh
```

После запуска сервер будет доступен по адресу:

```text
http://192.168.0.159:8080
```

Страница входа:

```text
http://192.168.0.159:8080/login
```


### 6.6 Дополнительный доступ с Windows-хоста

Основной сценарий остаётся таким же: веб можно открыть прямо на серверной Ubuntu ВМ.

Дополнительно, если основная система у вас Windows и она находится в той же домашней сети, web UI можно открыть и с хоста:

```text
http://192.168.0.159:8080/login
```

Это не заменяет основной Ubuntu-сценарий, а просто даёт ещё одну удобную точку входа для демонстрации.
### 6.6 Проверьте, что сервер отвечает

Откройте второй терминал на серверной ВМ и выполните:

```bash
cd /opt/barsuksiem/server-pack
./check-server.sh
```

Если все хорошо, вы увидите ответ от `/health`.

- делает это каждые 60 секунд.

Останавливается агент сочетанием клавиш `Ctrl+C`.

## 9. Как правильно запускать систему в первый раз

Правильный порядок такой:

1. Сначала серверная ВМ:
   - `cd /opt/barsuksiem/server-pack`
   - `./run-server.sh`
2. Потом ВМ аудитора:
   - открыть `http://192.168.0.159:8080/login`
   - войти как `admin / admin123`
3. Потом ВМ агента:
   - `cd /opt/barsuksiem/agent-pack`
   - `./run-agent.sh`
4. Потом снова на ВМ аудитора проверить:
   - `/logs`
   - `/incidents`
   - `/analytics`

## 10. Что проверять после запуска

### 10.1 Проверка сервера

Должно работать:
- `http://192.168.0.159:8080/health`
- `http://192.168.0.159:8080/login`

### 10.2 Проверка входа

После входа `admin / admin123` должны открываться страницы:
- `/`
- `/logs`
- `/incidents`
- `/analytics`

### 10.3 Проверка агента

После запуска агента на странице `/logs` должны начать появляться события.

Если среди событий есть подходящие шаблоны, они будут видны на странице `/incidents`.

### 10.5 Проверка аналитики

На странице `/analytics` должна появляться статистика.

## 12. Как поменять IP сервера

Если сервер у вас не `192.168.0.159`, нужно изменить только 2 файла:

На агентной ВМ:
- `/opt/barsuksiem/agent-pack/run-agent.sh`

На ВМ аудитора:
- `/opt/barsuksiem/auditor-pack/run-auditor-gui.sh`


на ваш реальный адрес сервера.

## 13. Типовые проблемы и решения

### Проблема: `/login` не открывается

Проверьте:
- запущен ли сервер через `./run-server.sh`;
- правильный ли IP-адрес сервера;
- открыт ли порт `8080` между ВМ;
- отвечает ли `http://192.168.0.159:8080/health`.

### Проблема: агент не отправляет логи

Проверьте:
- работает ли сервер;
- проходит ли `./check-agent.sh`;
- есть ли на Ubuntu журналы, которые агент может прочитать;
- не изменяли ли вы адрес сервера в `run-agent.sh`.

### Проблема: desktop-клиент не запускается

Проверьте:
- запускали ли `./install.sh` на ВМ аудитора;
- есть ли графическая среда Ubuntu;
- доступен ли сервер по сети.

### Проблема: в web UI пусто

Проверьте:
- запущен ли агент;
- подождите 60 секунд после старта агента;
- обновите страницу `/logs`.

## 14. Что не нужно делать в дипломной версии

В базовом сценарии не нужно:
- вручную создавать `env`-файлы;
- вручную выбирать пароль администратора;
- вручную разбирать проект на отдельные файлы;
- вручную ставить команды Python по одной;
- настраивать TLS/HTTPS;
- переносить SQLite-базу на агент или к аудитору.

Основной вариант для стенда:

## Demo-режим для защиты

Если нужен заранее подготовленный и полностью воспроизводимый стенд, используйте отдельный demo-режим сервера.

### Запуск demo-сервера

```bash
cd /opt/barsuksiem/server-pack
./run-server-demo.sh
```

Этот режим использует отдельную базу:

```text
/opt/barsuksiem/server-pack/data/demo.db
```

### Загрузка demo-данных

В отдельном терминале на серверной Ubuntu ВМ:

```bash
cd /opt/barsuksiem/server-pack
./load-demo.sh
```

После загрузки в системе уже будут доступны все 5 сценариев инцидентов.

### Полный reset demo-стенда

Самый удобный вариант перед защитой:

```bash
cd /opt/barsuksiem/server-pack
./reset-demo.sh
```

Скрипт делает всё автоматически:
- останавливает старый demo-сервер;
- удаляет `data/demo.db`;
- поднимает новый demo-сервер;
- загружает demo-набор;
- оставляет стенд в готовом состоянии.

### Как показывать на защите

Рекомендуемый порядок:
1. `./reset-demo.sh`
2. вход в web UI
3. dashboard `/`
4. список инцидентов `/incidents`
5. 2-3 карточки расследования `/incidents/details`
6. исходные логи `/logs`
7. короткий показ desktop-клиента как резервной витрины

Основной доступ остаётся на Ubuntu ВМ.
Дополнительно тот же интерфейс можно открыть с Windows-хоста по адресу:
