# Ревью кода и текущее состояние техдолга

## Что уже закрыто

В ходе последних доработок были устранены основные замечания высокого приоритета:

- убран hardcoded admin `admin/Admin123!@#`
- канонический API переведен на `/api/...`
- desktop-клиент переведен на защищенные API-маршруты
- CSRF flow унифицирован через `X-CSRF-Token` и `/api/auth/me`
- исправлена инвалидация кэша логов и инцидентов
- `shell=True` убран из агентской части
- сервер разрезан на route-модули
- добавлены parser / incidents / API tests
- README и проектная документация приведены к актуальному состоянию

## Что считается текущим техдолгом

### Frontend

- в проекте еще есть legacy-крупные скрипты, не подключенные в актуальных HTML страницах
- часть UI-файлов и HTML-шаблонов содержит старые битые текстовые фрагменты и требует отдельного редакторского прохода
- для frontend нет автоматических browser-side тестов

### Backend

- SQLite подходит для дипломной версии, но ограничен для дальнейшего масштабирования
- rule-based incident detection полезен для демонстрации, но ограничен по глубине корреляции
- role model пока минимальна: `admin`, `auditor`, `guest`

### Эксплуатация

- bootstrap password в startup log удобен для demo, но не является production-практикой
- для production нужна обязательная HTTPS-конфигурация и более строгая операционная модель управления секретами

## Что не считается критическим блокером защиты

- отсутствие сложной очереди ingest
- отсутствие PostgreSQL
- отсутствие ML/behavioral analytics
- отсутствие полноценной multi-user RBAC модели

Для дипломной версии это допустимо, если в отчете и на защите четко обозначить границы решения и направления развития.
